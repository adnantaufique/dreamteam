# Dreamteam ‚Äî Core Engine (the per-workstream loop)

Given an approved plan + a dreamteam manifest, drive each **workstream** to a verified, integrated result. The main loop is the **conductor** ‚Äî it stays visible and reports every gate verdict.

## The loop (per workstream)
```
produce   ‚Üí dispatch the producer agent(s) at their manifest tier ‚Äî resolve the dispatch verb + concrete model per references/platforms.md (Claude: Agent tool model:<resolved>; Gemini: @agent frontmatter model:; Codex: agent model=), with their manifest skills
gate      ‚Üí review per references/gate.md (parallel panel ‚Üí synthesize ‚Üí verdict) ‚Äî the dispatched panel is the workstream's risk-scaled reviewer subset (gate_policy.risk_scaling, references/caster.md)
fix       ‚Üí if fix-then-pass: run gate.md's capped fix loop (gate_policy.max_fix_iterations)
integrate ‚Üí land the verified artifact via superpowers:finishing-a-development-branch (merge / PR / keep ¬∑ then clean up the worktree) ‚Äî write the commit per ¬ß"Commit-message standard" below
report    ‚Üí print the workstream's verdict + the evidence behind the pass
```
- **Re-anchor every workstream (mandatory).** Before producing WS-N (including the **first** producer/specialist dispatch of the run), announce: **"Dispatching `<role>` for WS-N at `<tier>`, risk `<low|standard|high|off>` ‚Äî `<background subagents | Workflow>` (session mode)."** This line is required for *every* workstream ‚Äî it re-states the conductor rule and makes a missing dispatch an obvious tell (per `SKILL.md` ¬ß"You are the conductor"); the mode tag makes a foreground/inline slip as visible as a missing dispatch; the risk tag prints the class the conductor assigned at dispatch (`gate_policy.risk_scaling` ‚Üí the gate's reviewer subset, `references/caster.md`) ‚Äî when scaling is off or the panel is pinned (`--full-gate`, `--roster reviewers=‚Ä¶`, `--cost quality`), print `risk off`. (When a Workflow script drives the pipeline, the equivalent is the script's per-stage dispatch record ‚Äî the requirement is that every workstream's production is a *logged dispatch*, not that you type the line by hand.) If dispatch is unavailable, say so and pause ‚Äî never silently produce inline. Every unit dispatched is a **LEAF** (`SKILL.md` `<SUBAGENT-STOP>`): its brief carries the leaf-firewall (`references/caster.md`), so it does its one task and returns ‚Äî it never orchestrates or spawns further agents.
- **Pre-edit self-check.** Before any Edit/Write/Bash that would PRODUCE a workstream's artifact, ask: *"am I the producer for this workstream?"* If yes ‚Üí STOP and dispatch. The conductor edits only to **integrate** a verified result, never to produce one.
- **Integrate only after a pass** (or a fix-then-pass that cleared). A green gate is the entry condition for landing.
- **Verify before claiming done.** Before reporting a workstream ‚Äî or the whole run ‚Äî complete/passing, apply REQUIRED: `superpowers:verification-before-completion` (run the checks and cite the evidence; never assume green). It is the gate's honesty rule at the conductor level.
- **Execution discipline (producer/reviewer).** When a dispatched unit runs a shell command it MUST watch it to completion and READ its output + exit status BEFORE proceeding ‚Äî never proceed on an unobserved or unfinished command (this is the operational half of the emit-time evidence rule, `gate.md` ¬ß3: no observed output = no evidence = not done). For **long-running / backgrounded** commands use a **bounded** wait/poll (a timeout), then check the result ‚Äî **never** sit in an open-ended wait. If a command hangs past the bound: kill it and retry ONCE, or report the hang and continue with the evidence in hand ‚Äî **never** abandon the assigned task sitting in a wait state. A workstream is not "done" until its command evidence is actually captured. This is a **discipline** rule (model-compliance, like the leaf firewall) ‚Äî the harness owns the actual shell/poll; this is the model's obligation to observe it.
- **needs-work** from the gate ‚Üí stop, escalate, do **not** integrate and do **not** start dependents.
- **Designers** (manifest `designers[]`, when present) are scheduled as producers of the relevant design workstream ‚Äî the same produce‚Üígate‚Üíintegrate loop applies. (Profiles already express design as a producer role.)
- **Dispatch each producer with an explicit, verifiable success criterion** (what the gate will check) ‚Äî not "make it work." Before producing, **surface assumptions/ambiguities**; if genuinely under-specified, return **NEEDS_CONTEXT** rather than guessing (surfacing is pre-work; NEEDS_CONTEXT is its escalation).
- **Produce the simplest solution that fully works (minimal-code ‚Äî dreamteam's standing principle).** Every producer applies the ladder: *does it need to exist? (YAGNI) ‚Üí stdlib ‚Üí native platform feature ‚Üí an existing dependency ‚Üí one line ‚Üí the minimum that works*. Complete and correct, but no over-engineering / speculative generality ‚Äî and **never** by cutting validation, security, or accessibility ‚Äî and the **smallest diff that achieves it**: touch only what the task requires; don't refactor or restyle untouched code; remove only the orphans your change creates. The gate enforces this; the `ponytail` skill amplifies it when composed.

## Commit-message standard (integrate)
When the conductor lands a verified artifact (`integrate`), it **writes each commit for an OUTSIDE reader** ‚Äî a collaborator browsing the public history who never saw the plan, the run, or the gate. The message describes the **change**, not how the work was scheduled internally.
- **Subject** ‚Äî a conventional-commit line: `type(scope): imperative summary of WHAT changed` (e.g. `feat(auth): add token-refresh endpoint`), using the usual types (`feat|fix|docs|refactor|test|chore|‚Ä¶`).
- **Body (when useful)** ‚Äî the WHY/context: the problem solved, a constraint, a trade-off. Optional for a self-evident change; add it when the diff alone doesn't explain the motivation.
- **NEVER internal labels.** No build-phase names (`Phase A`/`Phase B`), no plan/checklist IDs (`P0-3`, `X-‚Ä¶`), no run/workstream numbers (`WS2`), no scenario IDs as the substance of the message ‚Äî these name dreamteam's internal scaffolding and are meaningless (and confusing) to an outside reader of the repo.
- **SELF-CHECK before committing.** Read the message back as someone who never saw the plan: *does it describe the change itself?* If it instead names an internal artifact (a build phase, plan/checklist ID, workstream/run number, or scenario), **rewrite it** to say what the diff does. This is a conductor self-check ‚Äî **not** a new gate and **not** a mandatory per-commit reviewer; the produce‚Üígate‚Üíintegrate contract is unchanged. (For repos with strict commit conventions, an optional `Git Workflow Master` reviewer MAY vet the message against the diff ‚Äî `references/recommend.md`, recommend-only.)

## Model tier + escalation (replicates subagent-driven-development)
Each producer is dispatched at its manifest tier and reports one status: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED.
- Gate **needs-work**, fix-then-pass not clearing, or **BLOCKED (needs more reasoning)** ‚Üí re-dispatch the fix/redo ONE TIER UP (cheap‚Üístandard‚Üícapable[‚Üímax only if the ceiling allows, per the cap below]), within `gate_policy.max_fix_iterations`. Never silently retry the same tier on a reasoning failure.
- **NEEDS_CONTEXT** ‚Üí add the missing context, re-dispatch SAME tier.
- **Task too large** ‚Üí split into smaller workstreams (don't just escalate the model).
- **Plan/spec wrong** ‚Üí escalate to the human (don't burn tiers).
- Cap: escalation stops at the manifest ceiling (default capable; max only if the run/override allows) and at `max_fix_iterations` ‚Üí then escalate to the human; never weaken the gate to force a pass.
- Reviewers do not auto-escalate (already at the capable default); a reviewer that errors is replaced per existing resilience (retry once first ‚Äî ¬ß"Dispatch-failure recovery"; then replace).
- **Tier unavailable / not dispatchable** ‚Üí fall back to the nearest available tier and flag it in the report (mirrors agent-unavailable in `SKILL.md` Resilience); never fail a workstream merely because a tier is missing.
- Report each escalation ("WS2 failed gate at standard ‚Üí retrying at capable").

## Sequencing (from `workstream_strategy`)
- **`sequential`** ‚Äî run workstreams in plan order; a dependent workstream waits for its predecessor to integrate.
- **`parallel`** ‚Äî run independent workstreams concurrently (e.g. ai-research *expand* ‚à• *polish*). Only parallelize workstreams with no shared inputs/outputs.

## Producer isolation (parallel work)
A producer that **mutates files** while other producers ‚Äî or the conductor prepping the next workstream ‚Äî run concurrently **gets its own git worktree**. REQUIRED: `superpowers:using-git-worktrees`. On a passing gate, land and clean up the worktree via `superpowers:finishing-a-development-branch`.
> Lesson from the build this skill generalizes: a producer and the conductor sharing **one** working tree forced everything to serialize. Separate worktrees = real parallelism without collision.

Read-only work needs no worktree. Verifying reviewers that contend for a build resource follow the same isolation rule (`gate.md` ¬ß1).

## Graph (conductor infra ‚Äî optional, AST-only, never a verdict)
An optional **AST code graph** (`graphify`; flag `--graph on|off|auto`, default `auto`) the conductor keeps as **navigation infra**. It is **NOT a workstream** ‚Äî it never appears in `producers[]`, is never gated, and **never decides a verdict**. AST-only (deterministic, no LLM ‚Üí free to build and refresh); graphify's semantic/LLM extraction is **not** used. The conductor maintains it in two steps:

- **G0 ‚Äî init at run start.** Once, before the fan-out: if `--graph` is `on`/`auto` **and** `graphify` is installed **and** the repo isn't oversized (graphify's own corpus warning) ‚Üí **presence-check ‚Üí reuse if fresh ‚Üí else build/refresh** the AST graph (`graphify <repo>`; an existing `graphify-out/graph.json` is reused, `graphify --update` refreshes only changed files). **Skip-if-absent:** `graphify` missing, `--graph off`, or an oversized repo ‚Üí **skip silently and continue** (the Caster may record a `recommendations[]` entry, `references/recommend.md`). The run **never blocks or fails** on the graph, and graphify is **never auto-installed**.
- **G1 ‚Äî refresh post-integrate.** After each workstream `integrate`s, refresh the graph as a `graph-update` sub-step (`graphify --update`, or the post-commit git-hook on the main tree ‚Äî `references/platforms.md`). Code-only changes are AST-only, so this is **free**. It is a refresh of infra, **not** a gated step: it runs after integrate, never blocks it, and a failed refresh is a warning, not a workstream failure. (The `audit` profile is the exception ‚Äî the graph is built **once at fan-out start** and `integrate` is a no-op, so there is **no G1**; see `references/audit.md`.)

**Why it helps (mechanism, not a number).** Producers and reviewers navigate one compact, persistent index ‚Äî communities, god nodes, shortest-path between symbols ‚Äî instead of repeatedly re-reading and re-grepping the tree to rebuild the same mental model each time; the graph persists across workstreams and sessions. The benefit is **fewer repeated file re-reads/re-greps and one shared map**, and how much it helps **depends on the repo size and the task** ‚Äî no token-savings percentage is claimed.

**CRITICAL invariant ‚Äî the graph never decides a verdict.** The graph is a *map*, not a *judge*. Any graph-derived claim that would gate a workstream (a bug, a dependency, an impact, an "unused" symbol) is an **unconfirmed candidate** until it is **verified against live code/tests** by the producer and the gate (`references/gate.md` ¬ß2, evidence beats prediction; graphify's own `INFERRED`/`AMBIGUOUS` edges are hints, not facts). A pass/fail verdict always comes from live evidence, never from the graph.

## Audit (read-only review/audit runs)
When the run is an `audit` profile, the loop still applies but **`integrate` is a no-op** ‚Äî the report **is** the artifact. A `needs-work` finding is **dropped from the report** (the issue it names is omitted), not cause to discard the whole deliverable. Audit producers are read-only, so they take **no worktree** (per ¬ß"Producer isolation" above ‚Äî same rule, not a new one). Even read-only finding-emitters are **dispatched subagents**: the conductor **never "just looks"** at the code and reports inline ‚Äî that would reintroduce the un-gated self-review the dispatch rule exists to prevent. See `references/audit.md` for the audit modes, `--depth`, and crew detail.

## Setup (opt-in pre-flight install)
On a human-**approved** recommendation (per `references/recommend.md`), the conductor dispatches a `setup` producer **before** the main crew ‚Üí the gate verifies presence + identity/source + that the dispatched install command was the pinned approved one (no `-y`) ‚Üí on PASS the conductor **re-casts** (re-enters precedence) so the now-installed skill attaches (re-cast at most once per approved set; a re-cast must not generate a new recommendation for the same set) ‚Üí FAIL ‚Üí escalate, proceed with best-available. Reuses produce‚Üígate‚Üíintegrate; no new gate machinery. The Caster only recommends ‚Äî **discovery in, advice out**; the `setup` producer installs, never the conductor inline.

## Leaves never orchestrate (the depth-1 invariant)
A dispatched producer / reviewer / refuter / planner / designer / Caster / setup agent is a **LEAF** (`SKILL.md` `<SUBAGENT-STOP>`): it may **NOT** call the Agent tool, the Skill tool, `/dreamteam`, or a Workflow script to spawn further agents ‚Äî it does its one briefed task and returns. **Only the conductor dispatches.** This has to be stated because nothing else forbids it: a dispatched unit *can* mechanically invoke Agent / Skill / Workflow, and there is **no run-wide dispatch tally** that would otherwise catch a leaf that fans out. Producer isolation (¬ß"Producer isolation") bounds *worktrees*, not dispatch depth ‚Äî **this** bounds dispatch depth, to the depth-1 invariant (`SKILL.md`, `run_policy.max_depth: 1`).

## Driving the loop: conductor vs Workflow tool
- **Dispatch is always background ‚Äî never inline/foreground in the main chat.** Producers/reviewers run detached (on Claude Code: the Agent tool's background dispatch / `run_in_background`); the conductor does NOT hand-fire blocking foreground Agent calls inline (the repeated drift this guards against). Background dispatch is non-blocking-in-chat, **NOT** fire-and-forget: the conductor still **awaits each producer's result, then gates it** (`gate.md`). The sequence dispatch‚Üíawait‚Üígate‚Üíintegrate is unchanged.
- **Execution mode ‚Äî chosen once per session (default background).** On the **first producer/specialist dispatch of the run**, ask the user once: "Background subagents or the Workflow tool? [background]". Use that answer for **every dispatch this SESSION**; surface it in the first re-anchor line. No answer ‚Üí **background subagents**. Ask **only once**; best-effort across the session (if the choice isn't in view later, re-ask rather than assume) ‚Äî never re-prompt per workstream. **When `--execution <mode>` is given on invocation, it PRE-SETS the session mode and SKIPS this prompt** ‚Äî the flag IS the answer (`background` = background subagents; `workflow` = the Workflow tool, Claude-Code-only per `platforms.md`); still surface the chosen mode in the first re-anchor line. With no flag, ask as today. This only pre-sets the one-time session choice ‚Äî dispatch stays always-background and there is no per-workstream re-prompt.
- **The Workflow tool (Claude Code only).** On Claude Code the deterministic `produce ‚Üí gate ‚Üí fix` pipeline MAY instead be a `Workflow` script (pipeline/parallel stages, a worktree per file-mutating producer) ‚Äî itself a background orchestration engine; prefer it for many independent workstreams. On Codex/Gemini/CodeWhale/OpenCode there is no Workflow tool ‚Üí background subagents always, no prompt (`platforms.md`).
- **Dispatch-failure recovery (both modes).** A dispatched unit that **errors or drops without returning** ‚Äî a harness error, the structured-output/schema retry cap in Workflow mode, a provider rate limit ‚Äî is re-dispatched **ONCE, at the SAME tier, with the SAME brief**. This is an **infrastructure error, not a reasoning failure**: no tier bump ‚Äî it is the error path that makes "a reviewer that errors is replaced per existing resilience" (¬ß"Model tier + escalation") concrete, and it is **distinct from** that section's needs-work/BLOCKED escalation (which re-dispatches ONE TIER UP). **Workflow mode:** a unit dropped at the structured-output retry cap is re-run **schema-free** ‚Äî for a reviewer that means plain-text findings + verdict the conductor parses; the schema is a convenience, never worth losing a unit over. **Partial stage failure** (some agents returned, some dropped/rate-limited) ‚Üí prefer **resume** (the Workflow tool's `resumeFromRunId`: completed agents replay from cache, only the dropped ones re-run) over re-running the whole stage fresh. An unreturned/dropped reviewer **NEVER counts as a pass** and its gate slot is **never skipped** ‚Äî recover it or escalate (a *returned-but-empty* response is handled by `gate.md` ¬ß"Verdict admissibility", which routes it back here). The recovery budget is **one re-dispatch per failure class per unit** ‚Äî dropped-without-returning (this bullet) and returned-but-inadmissible (`gate.md` ¬ß"Verdict admissibility") ‚Äî so at most **two** recovery re-dispatches per unit per gate. Every re-dispatch **counts toward `run_policy.max_total_dispatches`** ‚Äî the same stateless tally below, no new state; a unit still failing after its class's one retry ‚Üí replace the unit or escalate (`SKILL.md` ¬ßResilience).
- Either way the conductor **reports each workstream's gate verdict** ‚Äî orchestration never hides a gate.
- **The conductor honors `run_policy`** (`references/caster.md`): a run-wide concurrency cap (`max_concurrent_agents` ‚Äî serialize the excess rather than fan wider) and a cumulative `max_total_dispatches` backstop (**STOP + escalate to the human on reaching it**, never silently continue), tracked as a **simple integer in the stateless ¬ßReport buffer ‚Äî not a persisted store**. A large projected fan-out trips `budget_confirm_at` ‚Üí budget-print + confirm before producing (`caster.md` ¬ßAutonomy). These are conductor-side prose rules and hold on **all** CLIs, including the background-subagent path where no Workflow-harness limit exists (`references/platforms.md`); on Claude Code + the Workflow tool its own limits (where they apply) sit underneath as a second layer, but the caps do not rely on them.

## Report
Per workstream: the verdict (pass / fix-then-pass / needs-work), what was fixed, the **evidence** behind the pass, and what was integrated (commit / branch / artifact); when escalation occurred, the **tier path** (e.g. standard‚Üícapable). Pause only for a needs-work escalation or a genuine decision ‚Äî **classify every mid-run decision per `SKILL.md` ¬ßAutonomy (Mechanical / Taste / User-Challenge)**: Mechanical proceeds silently, Taste proceeds with the choice noted, a **User-Challenge** pauses (present the four-part frame, default to the user) even under `--autonomy auto`.

**Decision log (end-of-run ‚Äî STATELESS, report output only).** Close the run report with a short table of the conductor's key decisions ‚Äî roster/tier/scope picks, ambiguities resolved, each Taste call and any User-Challenge with its resolution (plus any noteworthy Mechanical call; routine Mechanical ones may be omitted). It is part of the printed report and of the ¬ßReport buffer the retro reads ‚Äî it is **NOT** a persisted or event-sourced store, **NOT** a JSONL/decision file, and **NOT** new state; dreamteam stays stateless by design. If no non-Mechanical decision was made, note "no notable decisions" instead. Columns:

| # | phase / workstream | decision | class | rationale | rejected alternative |
|---|---|---|---|---|---|
| 1 | _(e.g. cast)_ | _(what was decided)_ | Mechanical / Taste / User-Challenge | _(why)_ | _(what was not chosen)_ |

**Dispatch-reliability record (end-of-run ‚Äî same statelessness as the decision log).** The ¬ßReport buffer also carries a **per-role reliability tally**: drops by failure class (**dropped-without-returning**, ¬ß"Dispatch-failure recovery", vs **returned-but-inadmissible**, `gate.md` ¬ß"Verdict admissibility"), recovery re-dispatches, schema-cap **schema-free re-runs**, and **refuter outcomes** (**refuted-and-dropped** vs **stood**, `gate.md` ¬ß4), each tallied against the **emitting reviewer's row**, never a refuter row ‚Äî the refuter is the instrument; the reliability signal belongs to the prediction's **author** (which is what lets the retro spot a reviewer whose predictions are repeatedly refuted, `references/retro.md`). It is report output the retro reads ‚Äî **no new schema, NOT a persisted store, NOT a JSONL file**; dreamteam stays stateless.

**Context housekeeping (note, not a hook):** on a long, many-workstream run the conductor MAY `/compact` at a logical checkpoint ‚Äî a workstream boundary, after one workstream integrates/reports and before the next starts ‚Äî to keep its own context lean; it is an optional manual suggestion, never an automated step or a gate, and changes nothing in the produce‚Üígate‚Üíintegrate loop. End the run with the retro's learnings summary + any proposed skill-edit deltas (awaiting approval).

## Retro (run end)
After all workstreams integrate, run the retro when `--retro` is on (the default): `references/retro.md` over the run record (the ¬ßReport buffer ‚Äî per-workstream verdicts + escalation tier-paths + fix counts + the decision log + the dispatch-reliability record) ‚Üí persist learnings to `references/learnings.md` ‚Üí surface any proposed skill-edit deltas for human approval. **Skill edits are never auto-applied.** (`--evolve` additionally opts an ai-research run into the Layer-B generations loop, `references/evolve.md`.)
