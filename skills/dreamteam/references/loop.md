# Dreamteam — Core Engine (the per-workstream loop)

Given an approved plan + a dreamteam manifest, drive each **workstream** to a verified, integrated result. The main loop is the **conductor** — it stays visible and reports every gate verdict.

## The loop (per workstream)
```
produce   → dispatch the producer agent(s) at their manifest tier — resolve the dispatch verb + concrete model per references/platforms.md (Claude: Agent tool model:<resolved>; Gemini: @agent frontmatter model:; Codex: agent model=), with their manifest skills
gate      → review per references/gate.md (parallel panel → synthesize → verdict) — the dispatched panel is the workstream's risk-scaled reviewer subset (gate_policy.risk_scaling, references/caster.md)
fix       → if fix-then-pass: run gate.md's capped fix loop (gate_policy.max_fix_iterations)
integrate → land the verified artifact via superpowers:finishing-a-development-branch (merge / PR / keep · then clean up the worktree) — write the commit per §"Commit-message standard" below
report    → print the workstream's verdict + the evidence behind the pass
```
- **Re-anchor every workstream (mandatory).** Before producing WS-N (including the **first** producer/specialist dispatch of the run), announce: **"Dispatching `<role>` for WS-N at `<tier>`, risk `<low|standard|high|off>` — `<background subagents | Workflow>` (session mode)."** This line is required for *every* workstream — it re-states the conductor rule and makes a missing dispatch an obvious tell (per `SKILL.md` §"You are the conductor"); the mode tag makes a foreground/inline slip as visible as a missing dispatch; the risk tag prints the class the conductor assigned at dispatch (`gate_policy.risk_scaling` → the gate's reviewer subset, `references/caster.md`) — when scaling is off or the panel is pinned (`--full-gate`, `--roster reviewers=…`, `--cost quality`), print `risk off`. On a **micro-scale** task (`SKILL.md` spine step 2) the line additionally carries the scale declaration — `, task-scale micro: blast radius = <the one sentence>` — and **full battery prints no scale tag** (the default needs no declaring — leaner than printing `scale full` on every line). (When a Workflow script drives the pipeline, the equivalent is the script's per-stage dispatch record — the requirement is that every workstream's production is a *logged dispatch*, not that you type the line by hand.) If dispatch is unavailable, say so and pause — never silently produce inline. Every unit dispatched is a **LEAF** (`SKILL.md` `<SUBAGENT-STOP>`): its brief carries the leaf-firewall (`references/caster.md`), so it does its one task and returns — it never orchestrates or spawns further agents.
- **Pre-edit self-check.** Before any Edit/Write/Bash that would PRODUCE a workstream's artifact, ask: *"am I the producer for this workstream?"* If yes → STOP and dispatch. The conductor edits only to **integrate** a verified result, never to produce one.
- **Integrate only after a pass** (or a fix-then-pass that cleared). A green gate is the entry condition for landing.
- **Checkpoint discipline (`low` capacity).** Under a resolved `low` **capacity profile** (`references/caster.md` §"Capacity profile") the conductor **lands each workstream AS SOON AS its gate clears — never batching integrations, never holding a verified result un-landed while other workstreams continue** (a cadence rule, legal under the parallelism the `low` row still permits — sequential is only *biased*, `references/caster.md`): on a plan with tight session limits a session-limit kill must never lose a gated result — landing at gate-clear means a kill costs at most the in-flight work. Recovery after a kill is **resume-over-rerun** — pick up from the last integrated workstream and re-dispatch only what never completed (§"Dispatch-failure recovery"'s resume posture at run scale); already-gated, integrated work is never re-produced or re-reviewed.
- **Verify before claiming done.** Before reporting a workstream — or the whole run — complete/passing, apply REQUIRED: `superpowers:verification-before-completion` (run the checks and cite the evidence; never assume green). It is the gate's honesty rule at the conductor level.
- **Execution discipline (producer/reviewer).** When a dispatched unit runs a shell command it MUST watch it to completion and READ its output + exit status BEFORE proceeding — never proceed on an unobserved or unfinished command (this is the operational half of the emit-time evidence rule, `gate.md` §3: no observed output = no evidence = not done). For **long-running / backgrounded** commands use a **bounded** wait/poll (a timeout), then check the result — **never** sit in an open-ended wait. If a command hangs past the bound: kill it and retry ONCE, or report the hang and continue with the evidence in hand — **never** abandon the assigned task sitting in a wait state. A workstream is not "done" until its command evidence is actually captured. This is a **discipline** rule (model-compliance, like the leaf firewall) — the harness owns the actual shell/poll; this is the model's obligation to observe it.
- **needs-work** from the gate → stop, escalate, do **not** integrate and do **not** start dependents.
- **Designers** (manifest `designers[]`, when present) are scheduled as producers of the relevant design workstream — the same produce→gate→integrate loop applies. (Profiles already express design as a producer role.)
- **Dispatch each producer with an explicit, verifiable success criterion** (what the gate will check) — not "make it work." Before producing, **surface assumptions/ambiguities**; if genuinely under-specified, return **NEEDS_CONTEXT** rather than guessing (surfacing is pre-work; NEEDS_CONTEXT is its escalation).
- **Hand each dispatched unit its working set.** The brief carries the relevant file paths, entry points, (when the graph is on, §Graph) graph pointers, and the relevant MCP capability available on this host (the cast-time inventory, `references/caster.md` — availability only, never a dependency) — so the leaf spends its budget on the task, not on re-mapping the repo (the conductor-side counterpart of §Graph's shared-map rationale). The set is a head start, not a cage: a leaf MAY explore beyond it when the set proves insufficient — **saying so in its return** (that is reporting, never orchestrating — the leaf firewall is untouched); a genuinely insufficient set escalates as the existing **NEEDS_CONTEXT** (add the missing context, re-dispatch SAME tier — §"Model tier + escalation"). The status list there is producer-introduced but **unit-generic** under this rule: from a reviewer or refuter, **NEEDS_CONTEXT is a NON-VERDICT return** — handled by the same add-context + same-tier re-dispatch, **not** a verdict-admissibility failure (naming exactly what context is missing IS showing its work — `gate.md` §"Verdict admissibility") and **not** a dispatch failure (the unit returned; no recovery re-dispatch is consumed — §"Dispatch-failure recovery"). **The set's fuller form is the run context package** (`references/caster.md` — built ONCE per run, **facts only**, riding every brief's TAIL **headed by its fixed guard clause**, per the carriage format there): the conductor **merges each PRODUCER'S returned FACTS-DISCOVERED element (§"Model tier + escalation") into the package BETWEEN dispatches — never mid-dispatch** — and MAY fold in **off-map facts from ANY unit's return** (a reviewer's off-map finding names files too — the same merge, no new return element for reviewers) — so the next leaf inherits what the last one mapped instead of re-exploring for it. **Gate handoff:** a workstream's reviewers receive the package **plus the producer's touched-file list and the exact verify command it ran** — so they **VERIFY rather than re-map** — both labeled as **testimony**: the touched-file list is the producer's **CLAIM**, confirmed by `gate.md` §5's diff-review against the **ACTUAL DIFF** (the diff is ground truth; the list is an index into it), and the verify command is a **suggestion whose sufficiency the reviewer judges**; the package carries **facts, never conclusions** — a reviewer treats it as a map, not testimony: it still re-runs verification itself, and a finding is never inadmissible for being off-map (`gate.md` §1's independence guard; §3 unchanged).
- **Produce the simplest solution that fully works (minimal-code — dreamteam's standing principle).** Every producer applies the ladder: *does it need to exist? (YAGNI) → stdlib → native platform feature → an existing dependency → one line → the minimum that works*. Complete and correct, but no over-engineering / speculative generality — and **never** by cutting validation, security, or accessibility — and the **smallest diff that achieves it**: touch only what the task requires; don't refactor or restyle untouched code; remove only the orphans your change creates. The gate enforces this; the `ponytail` skill amplifies it when composed.

## Commit-message standard (integrate)
When the conductor lands a verified artifact (`integrate`), it **writes each commit for an OUTSIDE reader** — a collaborator browsing the public history who never saw the plan, the run, or the gate. The message describes the **change**, not how the work was scheduled internally.
- **Subject** — a conventional-commit line: `type(scope): imperative summary of WHAT changed` (e.g. `feat(auth): add token-refresh endpoint`), using the usual types (`feat|fix|docs|refactor|test|chore|…`).
- **Body (when useful)** — the WHY/context: the problem solved, a constraint, a trade-off. Optional for a self-evident change; add it when the diff alone doesn't explain the motivation.
- **NEVER internal labels.** No build-phase names (`Phase A`/`Phase B`), no plan/checklist IDs (`P0-3`, `X-…`), no run/workstream numbers (`WS2`), no scenario IDs as the substance of the message — these name dreamteam's internal scaffolding and are meaningless (and confusing) to an outside reader of the repo.
- **SELF-CHECK before committing.** Read the message back as someone who never saw the plan: *does it describe the change itself?* If it instead names an internal artifact (a build phase, plan/checklist ID, workstream/run number, or scenario), **rewrite it** to say what the diff does. This is a conductor self-check — **not** a new gate and **not** a mandatory per-commit reviewer; the produce→gate→integrate contract is unchanged. (For repos with strict commit conventions, an optional `Git Workflow Master` reviewer MAY vet the message against the diff — `references/recommend.md`, recommend-only.)

## Model tier + escalation (replicates subagent-driven-development)
Each producer is dispatched at its manifest tier and reports one status: DONE / DONE_WITH_CONCERNS / NEEDS_CONTEXT / BLOCKED. **A producer with attached `skills[]` adds ONE line to its return naming which attached skills it actually USED (and, briefly, to what effect); nothing to report → "attached skills unused"** — one return element, not a schema, carried into the run record so attachment decisions can improve (`references/retro.md`). **Every producer's return also carries ONE FACTS-DISCOVERED element (the same one-element pattern): the files it actually touched, the exact verify command it ran, and any surface found beyond the brief's map ("package deltas")** — facts only, never self-verdicts (whether the verify command *passed* is the gate's to establish, `gate.md` §3) — which the conductor merges into the run context package between dispatches (the working-set bullet above; `references/caster.md`).
- Gate **needs-work**, fix-then-pass not clearing, or **BLOCKED (needs more reasoning)** → re-dispatch the fix/redo ONE TIER UP (cheap→standard→capable[→max only if the ceiling allows, per the cap below]), within `gate_policy.max_fix_iterations`. Never silently retry the same tier on a reasoning failure.
- **NEEDS_CONTEXT** → add the missing context, re-dispatch SAME tier.
- **Task too large** → split into smaller workstreams (don't just escalate the model).
- **Plan/spec wrong** → escalate to the human (don't burn tiers).
- Cap: escalation stops at the manifest ceiling (default capable; max only if the run/override allows) and at `max_fix_iterations` → then escalate to the human; never weaken the gate to force a pass.
- Reviewers do not auto-escalate (already at the capable default); a reviewer that errors is replaced per existing resilience (retry once first — §"Dispatch-failure recovery"; then replace).
- **Tier unavailable / not dispatchable** → fall back to the nearest available tier and flag it in the report (mirrors agent-unavailable in `SKILL.md` Resilience); never fail a workstream merely because a tier is missing. Under a resolved **capacity profile** (`references/caster.md` §"Capacity profile") an expected substitution is declared **ONCE at cast — the manifest's capacity note — not re-flagged per dispatch**: the per-dispatch flag applies only to a **surprise beyond the declared profile** (a tier the profile treated as dispatchable failing anyway), which MAY also downgrade the profile once, flagged, per `caster.md`.
- Report each escalation ("WS2 failed gate at standard → retrying at capable").

## Sequencing (from `workstream_strategy`)
- **`sequential`** — run workstreams in plan order; a dependent workstream waits for its predecessor to integrate.
- **`parallel`** — run independent workstreams concurrently (e.g. ai-research *expand* ∥ *polish*). Only parallelize workstreams with no shared inputs/outputs.

## Producer isolation (parallel work)
A producer that **mutates files** while other producers — or the conductor prepping the next workstream — run concurrently **gets its own git worktree**. REQUIRED: `superpowers:using-git-worktrees`. On a passing gate, land and clean up the worktree via `superpowers:finishing-a-development-branch`.
> Lesson from the build this skill generalizes: a producer and the conductor sharing **one** working tree forced everything to serialize. Separate worktrees = real parallelism without collision.

Read-only work needs no worktree. Verifying reviewers that contend for a build resource follow the same isolation rule (`gate.md` §1).

## Graph (conductor infra — optional, AST-only, never a verdict)
An optional **AST code graph** (`graphify`; flag `--graph on|off|auto`, default `auto`) the conductor keeps as **navigation infra**. It is **NOT a workstream** — it never appears in `producers[]`, is never gated, and **never decides a verdict**. AST-only (deterministic, no LLM → free to build and refresh); graphify's semantic/LLM extraction is **not** used. The conductor maintains it in two steps:

- **G0 — init at run start.** Once, before the fan-out: if `--graph` is `on`/`auto` **and** `graphify` is installed **and** the repo isn't oversized (graphify's own corpus warning) → **presence-check → reuse if fresh → else build/refresh** the AST graph (`graphify <repo>`; an existing `graphify-out/graph.json` is reused, `graphify --update` refreshes only changed files). **Skip-if-absent:** `graphify` missing, `--graph off`, or an oversized repo → **skip silently and continue** (the Caster may record a `recommendations[]` entry, `references/recommend.md`). The run **never blocks or fails** on the graph, and graphify is **never auto-installed**.
- **G1 — refresh post-integrate.** After each workstream `integrate`s, refresh the graph as a `graph-update` sub-step (`graphify --update`, or the post-commit git-hook on the main tree — `references/platforms.md`). Code-only changes are AST-only, so this is **free**. It is a refresh of infra, **not** a gated step: it runs after integrate, never blocks it, and a failed refresh is a warning, not a workstream failure. (The `audit` profile is the exception — the graph is built **once at fan-out start** and `integrate` is a no-op, so there is **no G1**; see `references/audit.md`.)

**Why it helps (mechanism, not a number).** Producers and reviewers navigate one compact, persistent index — communities, god nodes, shortest-path between symbols — instead of repeatedly re-reading and re-grepping the tree to rebuild the same mental model each time; the graph persists across workstreams and sessions. The benefit is **fewer repeated file re-reads/re-greps and one shared map**, and how much it helps **depends on the repo size and the task** — no token-savings percentage is claimed.

**CRITICAL invariant — the graph never decides a verdict.** The graph is a *map*, not a *judge*. Any graph-derived claim that would gate a workstream (a bug, a dependency, an impact, an "unused" symbol) is an **unconfirmed candidate** until it is **verified against live code/tests** by the producer and the gate (`references/gate.md` §2, evidence beats prediction; graphify's own `INFERRED`/`AMBIGUOUS` edges are hints, not facts). A pass/fail verdict always comes from live evidence, never from the graph.

## Audit (read-only review/audit runs)
When the run is an `audit` profile, the loop still applies but **`integrate` is a no-op** — the report **is** the artifact. A `needs-work` finding is **dropped from the report** (the issue it names is omitted), not cause to discard the whole deliverable. Audit producers are read-only, so they take **no worktree** (per §"Producer isolation" above — same rule, not a new one). Even read-only finding-emitters are **dispatched subagents**: the conductor **never "just looks"** at the code and reports inline — that would reintroduce the un-gated self-review the dispatch rule exists to prevent. See `references/audit.md` for the audit modes, `--depth`, and crew detail.

## Setup (opt-in pre-flight install)
On a human-**approved** recommendation (per `references/recommend.md`), the conductor dispatches a `setup` producer **before** the main crew → the gate verifies presence + identity/source + that the dispatched install command was the pinned approved one (no `-y`) → on PASS the conductor **re-casts** (re-enters precedence) so the now-installed skill attaches (re-cast at most once per approved set; a re-cast must not generate a new recommendation for the same set) → FAIL → escalate, proceed with best-available. Reuses produce→gate→integrate; no new gate machinery. The Caster only recommends — **discovery in, advice out**; the `setup` producer installs, never the conductor inline.

## Leaves never orchestrate (the depth-1 invariant)
A dispatched producer / reviewer / refuter / planner / designer / Caster / setup agent is a **LEAF** (`SKILL.md` `<SUBAGENT-STOP>`): it may **NOT** call the Agent tool, the Skill tool, `/dreamteam`, or a Workflow script to spawn further agents — it does its one briefed task and returns. **Only the conductor dispatches.** This has to be stated because nothing else forbids it: a dispatched unit *can* mechanically invoke Agent / Skill / Workflow, and there is **no run-wide dispatch tally** that would otherwise catch a leaf that fans out. Producer isolation (§"Producer isolation") bounds *worktrees*, not dispatch depth — **this** bounds dispatch depth, to the depth-1 invariant (`SKILL.md`, `run_policy.max_depth: 1`).

## Driving the loop: conductor vs Workflow tool
- **Dispatch is always background — never inline/foreground in the main chat.** Producers/reviewers run detached (on Claude Code: the Agent tool's background dispatch / `run_in_background`); the conductor does NOT hand-fire blocking foreground Agent calls inline (the repeated drift this guards against). Background dispatch is non-blocking-in-chat, **NOT** fire-and-forget: the conductor still **awaits each producer's result, then gates it** (`gate.md`). The sequence dispatch→await→gate→integrate is unchanged.
- **Execution mode — chosen once per session (default background).** On the **first producer/specialist dispatch of the run**, ask the user once: "Background subagents or the Workflow tool? [background]". Use that answer for **every dispatch this SESSION**; surface it in the first re-anchor line. No answer → **background subagents**. Ask **only once**; best-effort across the session (if the choice isn't in view later, re-ask rather than assume) — never re-prompt per workstream. **When `--execution <mode>` is given on invocation, it PRE-SETS the session mode and SKIPS this prompt** — the flag IS the answer (`background` = background subagents; `workflow` = the Workflow tool, Claude-Code-only per `platforms.md`); still surface the chosen mode in the first re-anchor line. With no flag, ask as today. This only pre-sets the one-time session choice — dispatch stays always-background and there is no per-workstream re-prompt.
- **The Workflow tool (Claude Code only).** On Claude Code the deterministic `produce → gate → fix` pipeline MAY instead be a `Workflow` script (pipeline/parallel stages, a worktree per file-mutating producer) — itself a background orchestration engine; prefer it for many independent workstreams. On Codex/Gemini/CodeWhale/OpenCode/Cursor there is no Workflow tool → background subagents always, no prompt (`platforms.md`).
- **Dispatch-failure recovery (both modes).** A dispatched unit that **errors or drops without returning** — a harness error, the structured-output/schema retry cap in Workflow mode, a provider rate limit — is re-dispatched **ONCE, at the SAME tier, with the SAME brief**. This is an **infrastructure error, not a reasoning failure**: no tier bump — it is the error path that makes "a reviewer that errors is replaced per existing resilience" (§"Model tier + escalation") concrete, and it is **distinct from** that section's needs-work/BLOCKED escalation (which re-dispatches ONE TIER UP). **Workflow mode:** a unit dropped at the structured-output retry cap is re-run **schema-free** — for a reviewer that means plain-text findings + verdict the conductor parses; the schema is a convenience, never worth losing a unit over. **Partial stage failure** (some agents returned, some dropped/rate-limited) → prefer **resume** (the Workflow tool's `resumeFromRunId`: completed agents replay from cache, only the dropped ones re-run) over re-running the whole stage fresh. An unreturned/dropped reviewer **NEVER counts as a pass** and its gate slot is **never skipped** — recover it or escalate (a *returned-but-empty* response is handled by `gate.md` §"Verdict admissibility", which routes it back here). The recovery budget is **one re-dispatch per failure class per unit** — dropped-without-returning (this bullet) and returned-but-inadmissible (`gate.md` §"Verdict admissibility") — so at most **two** recovery re-dispatches per unit per gate. Every re-dispatch **counts toward `run_policy.max_total_dispatches`** — the same stateless tally below, no new state; a unit still failing after its class's one retry → replace the unit or escalate (`SKILL.md` §Resilience) — exception: a twice-failed **refuter** resolves as **no evidence either way → the finding STANDS**, never replace-or-escalate (`gate.md` §4).
- **Orphan-leaf watchdog (the conductor's await is bounded).** The conductor **BOUNDS every dispatched unit's wall-clock** — motivated by practice: a dispatched leaf once ran for many hours, silently awaited, because no bound existed. **Default bound: 3× the role's MEDIAN wall-clock this run** ("typical" pinned: the median of the per-role durations the §Report reliability record collects — one outlier never skews the bound), **never tighter than the absolute fallback below**; **no baseline yet** (e.g. the role's first dispatch) → fall back to a **stated absolute — 30 minutes** — and say so (a thin baseline is noisy; check-first + extend-once + doubt→escalate below already bound the damage). **Overridable per dispatch for known-long work** (a training run, a large benchmark sweep): state the expected bound in the dispatch brief at dispatch time — a declared-long dispatch does **not** trip the watchdog at the default bound; its stated bound IS the bound — **and the declaration is written to the run-scratch ledger at dispatch time (one line)**, so the compaction re-arm restores it (`SKILL.md` §Resilience): a re-armed conductor **never re-applies the default bound to a declared-long run** (the brief dies with context; the ledger doesn't). **On breach, CHECK FIRST:** is the unit demonstrably progressing (platform task status, output growth)? Demonstrably progressing → extend ONCE, stating the new bound. Not demonstrably progressing → **KILL + RE-DISPATCH ONCE** via §"Dispatch-failure recovery" (same tier, same brief, counted in the tally as a drop) **or ESCALATE** to the human; **in DOUBT about killing potentially-legit long work → ESCALATE, never destroy silently.** **In Workflow mode "kill" = the harness's task stop** — the Workflow tool's own limits sit underneath as ever, and the re-dispatch composes with resume-over-rerun (§"Dispatch-failure recovery" — cited, not restated). **On OpenCode the watchdog is POST-HOC — the degraded form, stated (the gate wave's announce pattern):** synchronous/modal dispatch means the conductor can ONLY await — a mid-flight check or kill is impossible (`references/platforms.md`, the OpenCode caveat) — so the bound is assessed when the call returns (the breach recorded in the reliability tally and reported), and the limitation is **stated, never papered over**. **NEVER silently await forever — everywhere the platform permits action** — this is the conductor-side sibling of the leaf's own §"Execution discipline" (the leaf never sits in an open-ended wait; the conductor never grants one).
- Either way the conductor **reports each workstream's gate verdict** — orchestration never hides a gate.
- **The conductor honors `run_policy`** (`references/caster.md`): a run-wide concurrency cap (`max_concurrent_agents` — serialize the excess rather than fan wider) and a cumulative `max_total_dispatches` backstop (**STOP + escalate to the human on reaching it**, never silently continue), tracked as a **simple integer in the stateless §Report buffer — not a persisted store**. A large projected fan-out trips `budget_confirm_at` → budget-print + confirm before producing (`caster.md` §Autonomy). These are conductor-side prose rules and hold on **all** CLIs, including the background-subagent path where no Workflow-harness limit exists (`references/platforms.md`); on Claude Code + the Workflow tool its own limits (where they apply) sit underneath as a second layer, but the caps do not rely on them. In Workflow mode with a **user token target**, the harness's live token budget is additionally honored per `run_policy`'s budget-aware scaling (`references/caster.md`) — degradation is **scheduling-only** (serialize → defer/shrink → STOP + escalate), **never the gate**; the static caps stay the floor, and the only layer everywhere else.

## Conductor economy (conductor-side speed — zero checks removed)
The why is measured in practice, not guessed: the conductor itself is the dominant wall-clock cost of a run — its own model-response gaps outweigh all agent execution combined, because it carries the run's largest context, answers in many small sequential steps, and re-digests every leaf's full narrative — and observed leaf parallelism stays near-sequential even though `gate.md` §1 mandates a parallel panel (the cost is what's observed — no savings figure is claimed, §Graph's posture). The rules below delete that duplicated work. **The non-negotiable is the run context package's guard applied to the conductor's own steps: speed comes ONLY from adding information and deleting duplicated work, NEVER from removing a check** (`references/caster.md`) — no gate, verdict rule, or honesty rule changes here.

- **Gate waves — §1's parallelism made operational.** The gate's panel dispatches as **ONE WAVE = ALL panel members eligible to run concurrently under `gate.md` §1's resource split** — static reviewers always, **verifying reviewers INCLUDED whenever uncontended** (a sole verifier contends with nothing: on the common 2-reviewer panel the Reality Checker rides the wave) — a single step of simultaneous background dispatches (one message, all the wave's Agent calls together), never one-reviewer-per-response — within `run_policy.max_concurrent_agents` (serialize the excess, §"Driving the loop"); **only a genuinely contended verifier serializes or isolates** (§1's rule, cited not restated), **and the announce SAYS which and why**. **The gate announce STATES the wave composition, TRUTHFULLY** — *"gate wave: Reality Checker + Code Reviewer, parallel"* — the same tell-making pattern as the re-anchor line: a panel that quietly went out one-at-a-time becomes a visible drift, never a silent one. **The obligation is "as parallel as the platform and caps permit, ANNOUNCED TRUTHFULLY":** where parallel dispatch is legitimately impossible — OpenCode's synchronous dispatch (`references/platforms.md`), a `max_concurrent_agents` cap smaller than the panel, budget serialization — the announce carries the degraded form and its reason — *"gate wave: Reality Checker + Code Reviewer, serialized — OpenCode synchronous dispatch"* / *"…, serialized — cap 3, two slots held by a concurrent workstream's producers"* / *"…, serialized — budget"* — and **printing ", parallel" falsely is itself a violation** (the no-glazing voice, `gate.md` §3, applied to the conductor's own announces). When workstreams are genuinely independent (`workstream_strategy: parallel`), the conductor MAY prep and dispatch the next workstream while another's gate wave runs — awaiting a wave is never idling the run. (When a Workflow script drives the pipeline, its parallel stages already are the wave — the obligation is the operational parallelism, not a hand-typed message.)
- **Batched conductor steps.** The conductor issues **INDEPENDENT tool calls together in one step** — the reads, greps, and status checks a single decision needs — never one-per-response: each extra round-trip is pure model-gap on the run's largest context. Dependent calls (one call's input needs another's output) stay sequential — batching never forces a dependency together.
- **Digest, don't ingest.** From each leaf return the conductor carries forward the **STRUCTURED CORE = EVERY element of the unit's RETURN CONTRACT**: the status · the **per-reviewer VERDICT + its shows-its-work / pass rationale VERBATIM** (`gate.md` §"Verdict admissibility" — on a zero-finding clean pass that rationale IS the entire evidence, and §2's `min_pass` consumes it) · the **FINDINGS VERBATIM — NEVER summarized, NEVER paraphrased: `gate.md` §2's synthesis consumes findings whole, and a paraphrased finding is a corrupted input, a violation, not a saving** · the FACTS-DISCOVERED element (§"Model tier + escalation") · the **skill-usage line** (§"Model tier + escalation" — retro input) · **evidence pointers** (paths to the full outputs on disk) — and **DIGESTS the narrative** (the leaf's journey/prose) to one line in the §Report buffer (a digest line in the run-scratch ledger below, when kept on disk). **The boundary is one step: an element of the return contract travels WHOLE; only free narrative OUTSIDE the contract is digested.** It re-reads a full output from disk only when synthesis genuinely needs it — the evidence pointer is what keeps that cheap. **Contract-verbatim is the capability guard: the digest applies to the narrative ONLY — nothing the verdict, the report, or the retro consumes is ever compressed**, so synthesis quality is untouched by construction.
- **Run-scratch ledger (per-run working memory — stateless across runs).** The §Report buffer lives in a **per-run scratch FILE — MANDATORY FROM THE FIRST UNIT RETURN of any run**: the decision log, the dispatch-reliability tally, the run context package's current state (`references/caster.md`), and the per-leaf digest lines, **appended as the run goes (ONE WRITE PER EVENT: a return's structured core + digest line as it lands, a decision as it's taken, a declared-long bound at dispatch)** — the ledger IS the report buffer, not a backup of it; "optional" survives only for a run that never dispatches. The always-current file is what lets the conductor `/compact` aggressively at workstream boundaries (§Report's housekeeping note) without losing run state, keeping its own context lean. **Where and how, by construction:** the ledger lives **OUTSIDE the repo working tree** (the platform's scratch/temp area — never workspace pollution, never something the package scout could stumble on), is **named with a unique per-run identifier**, and is **CREATED FRESH at run start — never opened-if-exists**: no run ever reads another run's ledger, so an abandoned ledger is inert garbage, never input. **Externalize-then-compact — the deliberate-compact restatement, no longer the sole guarantee:** before any chosen mid-run `/compact`, the structured cores + digest lines accumulated so far are already in the ledger (the write-per-event mandate above), so "re-read from disk" always has a referent. The mandate can't wait for intent because **a compaction is not always chosen: AUTO-compaction (context overflow) can strike a conductor that never planned to compact** — a ledger created only when compaction is intended would be a false guarantee for exactly that case. **Auto-compact recovery (the never-created / lost case, honestly):** when the re-arm finds NO ledger — a run that never dispatched, an auto-compact that struck before the first write, a lost file — recover from **disk ground truth**: landed commits, the working tree, the plan document, the run context package when the plan carries it, plus the compaction summary; the dispatch tally follows the ledger-loss rule below (last known count carried forward — continuing is a User-Challenge). Recovered state is honestly LESS than ledger state — the report says so, never papers over it. **Its read-back half is the compaction re-arm** (`SKILL.md` §Resilience): a conductor past a compaction boundary re-reads `SKILL.md` from its on-disk home AND this ledger before the next artifact-producing action — the first-unit-return mandate is what guarantees the ledger is there to re-read for any run past its first unit return (none there → auto-compact recovery above, which covers the dispatch→first-return window too), so a compact — chosen or automatic — costs neither run state nor rule fidelity. **Ledger loss never resets the run:** a ledger lost mid-run **never re-zeroes the `run_policy` dispatch tally** — the conductor carries the **last known count forward** (conservative, never re-zeroed) and treats continuing after a loss as a **User-Challenge** pause (`SKILL.md` §Autonomy). **Per-run lifetime, strictly:** created fresh at run start; the **printed report remains the artifact of record**; the file is **DELETED (or explicitly abandoned) at run end — only AFTER the printed report and the retro (§Retro) have consumed the buffer** — dreamteam stays **STATELESS ACROSS RUNS**. The ledger is the **same §Report buffer, on disk for the run's duration** — never a persisted or event-sourced store, never a cross-run file, never read by a later run (the retro's `references/learnings.md` remains the only sanctioned persistence) — and distinct from the *agent scouting ledger* there, which the retro persists; this one never survives the run.

## Report
Per workstream: the verdict (pass / fix-then-pass / needs-work), what was fixed, the **evidence** behind the pass, and what was integrated (commit / branch / artifact); when escalation occurred, the **tier path** (e.g. standard→capable). Pause only for a needs-work escalation or a genuine decision — **classify every mid-run decision per `SKILL.md` §Autonomy (Mechanical / Taste / User-Challenge)**: Mechanical proceeds silently, Taste proceeds with the choice noted, a **User-Challenge** pauses (present the four-part frame, default to the user) even under `--autonomy auto`.

**Statelessness umbrella (every §Report element).** Every buffer element in this section — the per-workstream verdict elements above, the decision log, the handoff notes, the dispatch-reliability record — shares ONE statelessness: **report output only**, transiently sitting in the per-run **run-scratch ledger** (§"Conductor economy" — mandatory from the run's first unit return) for the run's duration, **never anything that outlives the run** — stated once here so the notes below cite it rather than re-derive it.

**Plain-language verdict line (one per workstream — additive, default on, no flag).** Directly after the verdict elements above, print **one sentence in ordinary words** stating what the gate checked and what that means for this workstream — written for a reader who doesn't know the verdict vocabulary, shaped to what the gate actually checked here. The gloss is **faithful to `gate.md` §2's definitions and never rounded up** (the no-glazing / objective standard, `SKILL.md`, applies to plain language too): a **pass** reads like *"an independent reviewer re-ran the tests and confirmed the claims — safe to build on"*; a **fix-then-pass** reads like *"the reviewer found 2 real problems; they were fixed and re-checked — nothing shipped unreviewed"* — never softened into a clean pass; a **needs-work** reads like *"the approach itself failed review — nothing was accepted into the project, and a human decision is needed"*. **One sentence, never a paragraph.** The expert elements above are unchanged and remain the report of record — the gloss translates them, never replaces them.

**Decision log (end-of-run — STATELESS, report output only).** At run end, print a short table of the conductor's key decisions — roster/tier/scope picks, ambiguities resolved, each Taste call and any User-Challenge with its resolution (plus any noteworthy Mechanical call; routine Mechanical ones may be omitted). It is part of the printed report and of the §Report buffer the retro reads — it is **NOT** a persisted or event-sourced store, **NOT** a JSONL/decision file, and **NOT** new state; dreamteam stays stateless **across runs** by design (the statelessness umbrella above). If no non-Mechanical decision was made, note "no notable decisions" instead. Columns:

| # | phase / workstream | decision | class | rationale | rejected alternative |
|---|---|---|---|---|---|
| 1 | _(e.g. cast)_ | _(what was decided)_ | Mechanical / Taste / User-Challenge | _(why)_ | _(what was not chosen)_ |

**Handoff notes (end-of-run — plain language; same statelessness as the decision log).** After the decision log — closing the user-facing report — print **up to 5 plain bullets naming what still needs a HUMAN**: what to review manually (e.g. user-facing copy, visual checks the gate couldn't exercise), what remains **untested or manual-only** per the gate's proof-labeling (`gate.md` §3 — the handoff repeats that label honestly, never reporting a lower tier as a higher one), any debt or deferral noted mid-run (including a budget-pressure deferral, `references/caster.md`), and any security/config step the run couldn't do itself (secrets to set, a deploy to run). When nothing qualifies, print **"nothing outstanding — all workstreams verified"** — never silently omit the section. Like the decision log it is **part of the printed report only — NOT a persisted store, NOT new state**.

**Dispatch-reliability record (end-of-run — same statelessness as the decision log).** The §Report buffer also carries a **per-role reliability tally**: drops by failure class (**dropped-without-returning**, §"Dispatch-failure recovery", vs **returned-but-inadmissible**, `gate.md` §"Verdict admissibility"), recovery re-dispatches, schema-cap **schema-free re-runs**, **refuter outcomes** (**refuted-and-dropped** vs **stood**, `gate.md` §4), and **per-role WALL-CLOCK** (each dispatch's duration; the tally reports **median + max per role** — what makes "why was this run slow" answerable from the run record, and the baseline the orphan-leaf watchdog's 3× bound reads, §"Driving the loop") — **refuter outcomes alone** are tallied against the **emitting reviewer's row**, never a refuter row (drops, recovery re-dispatches, and schema-free re-runs stay on the **failing unit's own row** — that is what per-role means for them): the refuter is the instrument; the reliability signal belongs to the prediction's **author** (which is what lets the retro spot a reviewer whose predictions are repeatedly refuted, `references/retro.md`). It is **retro-facing** report output the retro reads — **no new schema, NOT a persisted store, NOT a JSONL file**; dreamteam stays stateless.

**Report proportionality (format scales with run size — content never).** A **1-workstream run reports on one screen — carrying THE SAME CONTENT SET as this section**: the verdict · its plain-language gloss · the evidence · what was fixed · what was integrated (commit / branch / artifact) · the tier path when escalation occurred · the handoff (up to 5 bullets, per its own rule) — **single-line renderings, nothing dropped**. The full table apparatus — the decision-log columns, the reliability tally — prints when the run is big enough to need it and otherwise **collapses to single lines**, extending the existing "no notable decisions" pattern (e.g. "no dispatch failures — no reliability tally to print"). **NOTHING is omitted — only the FORMAT shrinks:** every workstream still reports the full content set above, every element remains the report of record, and the gloss stays faithful — a small run prints them small, never skips them.

**Context housekeeping (note, not a hook):** on a long, many-workstream run the conductor MAY `/compact` at a logical checkpoint — a workstream boundary, after one workstream integrates/reports and before the next starts — to keep its own context lean (externalize-then-compact, §"Conductor economy": the run state is in the run-scratch ledger BEFORE any mid-run compact, so an aggressive compact at a boundary loses nothing the report needs); past any compaction boundary the **compaction re-arm** applies (`SKILL.md` §Resilience — re-read `SKILL.md` from disk + this ledger, re-anchor, before the next artifact action); it is an optional manual suggestion, never an automated step or a gate, and changes nothing in the produce→gate→integrate loop. End the run with the retro's learnings summary + any proposed skill-edit deltas (awaiting approval).

## Retro (run end)
After all workstreams integrate, run the retro when `--retro` is on (the default): `references/retro.md` over the run record (the §Report buffer — per-workstream verdicts + escalation tier-paths + fix counts + the decision log + the dispatch-reliability record + each producer's skill-usage line) → persist learnings to `references/learnings.md` → surface any proposed skill-edit deltas for human approval. On a **micro-scale** run (`SKILL.md` spine step 2) the retro is **dispatch-free**: the conductor appends the run record's learnings line to the ledger/report itself — no feedback-agent dispatch — and any learnings-store write still follows `references/retro.md`'s evidence rule (no learning without run evidence). **Skill edits are never auto-applied.** (`--evolve` additionally opts an ai-research run into the Layer-B generations loop, `references/evolve.md`.)
