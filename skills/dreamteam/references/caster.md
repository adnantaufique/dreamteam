# Dreamteam — The Caster (dreamteam selection)

Resolve the dreamteam for a task and **print a dreamteam manifest** (+ one line of rationale per pick) before any work runs. The Caster only *selects* — it never produces.

Skill recommendations + the opt-in installer role: `references/recommend.md` (the Caster only recommends — **discovery in, advice out**).

## Precedence (stop at the first that applies)
0. **Consult learnings** — read `references/learnings.md` for the matched profile/task_kind, **filtered by `project_key`**: consult **GLOBAL entries (`project_key` = `global`, or a blank/legacy cell) always + PROJECT entries whose `project_key` matches the current repo's git-remote hash** — a learning from another project never leaks in. Treat any matched entry as an overridable default **weighted by its numeric `confidence` (0.3–0.9)** (adjust the roster/tiers/skill attachments/gate emphasis accordingly — a row may carry the retro's skill-usage advice), never a hard rule. This informs — never overrides — the steps below. **Consult the agent scouting ledger the same way** (same file, same `project_key` filter): it informs the **choice between fitting candidates** — the Caster-agent path (3) and any tie-break where more than one installed agent fits a role — **preferring the better first-try / lower-drop record**; a row with **fewer than 3 dispatches is ignored as noise** (low-sample humility). Advisory only — **a row never bans an agent, and a MISSING row is never held against one** (a never-dispatched agent competes on fit exactly as before): it never overrides an explicit `--roster` (1), never removes a profile fast-path row's core agents (2 — only their *absence* breaks the fast path, unchanged), and **never touches the Reality Checker** (always-on regardless of any record). When the ledger influenced a pick, the rationale line **says so with the numbers** (e.g. "ledger: 7/8 first-try over 2/5").
1. **Explicit cast** — any of `--roster` / `--profile` / `--skills` given → build the manifest from them, skip the Caster agent.
   - `--profile X` → that row of `references/profiles.md`. Back-compat: `--profile android` resolves to `mobile-dev` (alias), and the learnings lookup key normalizes `android → mobile-dev` so persisted `references/learnings.md` rows still match after the rename.
   - `--roster planner=<agent|skill>[@<tier>],producers=<role>:<agent>[@<tier>][+<skill>…][;<role>:<agent>…],reviewers=<agent>[@<tier>][,<agent>…]` → parse into the manifest as given (Reality Checker is still force-added to `reviewers` if the user omitted it).
   - `--skills a,b` → force-include those skills, attaching them to the relevant producer(s)' `skills[]` (or to the sole producer when there is one).
   - `--models planner=<tier>,producers=<role>:<tier>;…,reviewers=<tier>` → set tiers explicitly.
   - `--cost cheap|balanced|quality` → bias the **producer** rubric (cheap: producers→cheap tier; balanced [default]: the rubric as written; quality: producers→standard/capable, max only here as an explicit opt-in for the hardest). **Reviewers never drop below capable — Reality Checker especially — regardless of `--cost`** (the honesty backbone stays strong).
   - `--full-gate` → set `gate_policy.risk_scaling: off` — the full reviewer panel on every workstream, **touching nothing else** (the dedicated off-switch: no `--cost quality` producer-tier raise, no hand-specified `--roster` panel).
   - `--roster …@<tier>` → an `@tier` suffix on any role sets that role's tier inline (e.g. `producers=build:Mobile App Builder@standard`; legacy Claude names like `@sonnet` are accepted via back-compat); a bare role keeps the rubric/profile default.
2. **Profile fast-path** — no explicit cast → try a confident profile match (the matching rules in `references/profiles.md`). Clean match → **check the row's core agents (producers + reviewers) are available** (the live agent list). All present → use that row's dreamteam, no agent hop. **A missing core agent → skip the fast-path and fall through to the Caster agent (3)**, which adapts to what is actually installed. **An OPTIONAL composed skill that's absent does NOT disqualify the fast-path:** the enforcers (`ponytail`, `karpathy-guidelines`), a profile's composed external skills (`mle-workflow`, `systematic-debugging`, `ui-ux-pro-max`), and the optional `graphify` navigation graph attach **only when installed** (`find-skills`/tool-gated, each checked independently); when absent the producer works natively and the Caster records a `recommendations[]` entry (`references/recommend.md`) — never a block, never a fall-through. A profile's **non-waivable gate dimension stands regardless** of a missing skill (e.g. ux-designer's a11y without `ui-ux-pro-max`).
3. **Caster agent** — novel / ambiguous / cross-domain / no clean profile → dispatch a `general-purpose` **Caster agent** (prompt below) to reason over the live registries and return a manifest.
   - If a materially-better skill is discoverable (`find-skills` / the awesome-claude-code CSV) but not installed, record it in `recommendations[]` (per references/recommend.md) and proceed with the best installed option — never block. Explicit `--roster/--skills` casts bypass this (a missing named skill is handled by the existing find-skills existence check).

Whichever path, **Reality Checker is always in `reviewers`** (the claim-integrity backbone).

## Dreamteam manifest — the cross-file contract
Defined here once; `loop.md`/`gate.md`/`profiles.md`/`audit.md` read these field names verbatim; unknown advisory fields (e.g. `recommendations[]`) are ignored, never required.
```
{ planner: { agent|skill, model } | null,        // model only when planner is a dispatched agent
  designers: [ { agent, model } ],                // scheduled as producers; each carries a tier
  producers: [ { role, agent, model, skills[] } ],
  reviewers: [ { agent, model } ],                // reviewers now objects; Reality Checker present + capable
  gate_policy: { min_pass, max_fix_iterations, devils_advocate_on_unanimous: false, devils_advocate_min_panel: 3, risk_scaling: on },
  audit_policy: { max_parallel_reviewers },       // sibling of gate_policy, NOT nested inside it
  run_policy: { max_depth: 1, max_concurrent_agents, max_total_dispatches, budget_confirm_at },  // run-wide caps, sibling of gate_policy/audit_policy
  workstream_strategy: "sequential" | "parallel",
  recommendations: [ ... ],                       // optional, advisory, per-run ephemeral — see references/recommend.md
  rationale: [ <one line per pick, naming the tier + why> ] }
```
- **`min_pass`** — reviewers that must reach a passing verdict for a workstream to integrate (default: all reviewers), counted **of the dispatched (risk-scaled) subset** when `risk_scaling` is on; an explicit **numeric `min_pass` is a FLOOR** — the conductor **widens** the workstream's subset until it satisfies it (never silently clamps `min_pass` down, never leaves a workstream unsatisfiable). Reality Checker's pass is *always* required regardless of `min_pass`.
- **`max_fix_iterations`** — cap on fix → re-verify cycles before escalating to the human (default: 2).
- **`devils_advocate_on_unanimous`** — opt-in (default: `false`); when `true`, a unanimous pass triggers one devil's-advocate re-review before integrating. The `audit` profile sets it `true`.
- **`devils_advocate_min_panel`** — minimum reviewer count for the devil's-advocate pass to fire (default: 3); smaller panels skip it.
- **`gate_policy.risk_scaling`** — `on|off` (default **on**): risk-proportional gate depth. When `on`, the conductor classifies **each workstream at dispatch** into a **risk class** — `low | standard | high` — and the gate dispatches a **risk-scaled SUBSET** of the manifest's `reviewers` panel: **low → Reality Checker only · standard → Reality Checker + the most relevant domain reviewer · high → the full panel**. The risk class is **NOT a manifest field** (workstreams live in the plan, not the manifest) — the conductor assigns it at dispatch and prints it in the re-anchor line (`loop.md`).
  - **Classification heuristics:** **diff scope** (files/lines touched) · **sensitive surface** (auth / payments / data / infra / a11y / security → `high`) · **novelty** (a new subsystem vs a mechanical change) · **claim exposure** (user-facing or depended-on results). **Any doubt → the higher class.**
  - **Guards — classes subtract, guards force-include.** **Reality Checker is ALWAYS dispatched and its pass ALWAYS required**, whatever the class; a reviewer covering an in-scope **non-waivable dimension** (a11y under ux-designer — `gate.md` §3; security when in scope for the workstream — `gate.md` §1) is **always dispatched in addition** to the class's base subset — belt-and-suspenders against **misclassification**: the force-include holds even when the conductor classed the workstream too low; `min_pass` applies to the **dispatched subset**; `--full-gate`, an explicit `--roster reviewers=…`, or `--cost quality` **PINS the full panel**; `risk_scaling: off` = today's behavior (the full panel on every workstream). Natural interaction: an RC-only `low` panel can never trigger the devil's-advocate pass **at the defaults** (its size 1 sits below the default `devils_advocate_min_panel: 3`).
  - **Panel SIZE, never tier — and never audit.** Risk scaling subsets the panel; it changes **no tier** — reviewers stay capable regardless (`--cost`'s "reviewers never drop below capable" is a TIER floor, orthogonal to this SIZE dial). Risk scaling governs the **produce→gate→fix loop's gate**; an `audit` run's reviewer fan-out is its own machinery, governed by `audit_policy` (`references/audit.md`) — **never by risk classes**.
- **`audit_policy.max_parallel_reviewers`** — cap on reviewers dispatched concurrently per workstream (top-level sibling of `gate_policy`, read by `audit.md`).
- **`run_policy`** — run-wide safety caps, a top-level sibling of `gate_policy` / `audit_policy`, read by `loop.md` (the conductor honors it when driving the fan-out). Reuses the same **two-tier "per-run default + hard ceiling nothing overrides"** shape as `audit_policy` (audit's ceiling-12 pattern), generalized from the reviewer pool to the whole run — the per-field defaults below are the **`high` capacity row**, resolved per run by the **capacity profile** (§"Capacity profile" below; the hard ceilings never move):
  - **`run_policy.max_depth`** — default **1**, and a **stated structural invariant**: the run is conductor → leaves, one level; a dispatched leaf never orchestrates (`SKILL.md` `<SUBAGENT-STOP>`). This is **not** a counter tracked across a call tree — the identity firewall guarantees it, needing zero shared state.
  - **`run_policy.max_concurrent_agents`** — live dispatched agents at any instant, across ALL workstreams. Default **8**, hard ceiling **16** that no profile/override exceeds. When parallel workstreams would exceed it, the conductor **serializes the excess rather than fanning wider** (the run-wide analogue of `audit_policy.max_parallel_reviewers`).
  - **`run_policy.max_total_dispatches`** — cumulative lifetime backstop for the whole run (counts producers + reviewers + fix **and recovery** re-dispatches + refuters (`gate.md` §4) + Caster + setup + retro). Default **60**. On reaching it the conductor **STOPS and escalates to the human — never silently continues**. Tracked as a light integer in the stateless §Report buffer (`loop.md`), **not** a persisted or event-sourced store.
  - **`run_policy.budget_confirm_at`** — projected-fan-out threshold that forces a confirm-gate. Default **30**. Before the fan-out the conductor estimates `projected ≈ Σ workstreams × (producers + panel) × (1 + max_fix_iterations)`; if `projected ≥ budget_confirm_at` (or `--depth exhaustive` / `--evolve` is set) it **prints the projected fan-out + cost and auto-enters the `confirm` step below (waits for the user's OK) even under `--autonomy auto`**. Generalizes the opt-in `exhaustive` / `--evolve` budget-prints to any large run — reuses the existing `confirm` machinery, adds none.
  - **Budget-aware scaling (Claude Code + Workflow mode only).** When the session carries a **user token target**, the conductor ALSO honors the Workflow harness's **live budget** — its remaining-token reading — as a **run ceiling alongside the static caps above**: the §Autonomy budget-print/projection incorporates it (the projected fan-out + cost is weighed against the tokens remaining, not only against `budget_confirm_at`'s dispatch count). On budget pressure the conductor **DEGRADES BY SCHEDULING ONLY**, in this order: **(a)** serialize parallel workstreams (less concurrency), **(b)** defer or shrink the remaining fan-out (fewer/smaller workstreams, postponed optional dispatches — **"optional" = a dispatch whose absence CANNOT change a verdict**: skipping a pre-fix refuter is fail-safe by construction (no evidence → the finding STANDS, `gate.md` §4), whereas an **ARMED devil's-advocate pass** (`gate_policy.devils_advocate_on_unanimous: true`, e.g. the `audit` profile) is **PART OF THE GATE — never skipped or postponed under budget pressure** (it belongs to the never-weaken family below); a deferral is reported, never a silent scope drop), **(c)** **STOP + escalate to the human, exactly like reaching `max_total_dispatches`**. **Budget pressure NEVER weakens the gate: it never reclassifies a workstream's risk, never shrinks a panel below its class, never drops the Reality Checker or a non-waivable-dimension reviewer, never lowers a tier floor, and never relaxes the honesty rules (`gate.md` §3) — a run that cannot afford its gates STOPS; it does not ship less-reviewed work.** The static caps above remain the FLOOR everywhere (the live budget only ever tightens a run beyond them, never loosens one), and the ONLY layer on background mode and the other four CLIs — no harness budget exists there (`references/platforms.md`). A user token target given **on those no-budget paths** is **ACKNOWLEDGED explicitly — the conductor states that no live budget exists on this path and the static caps govern — never silently ignored** (the conductor does not hand-track tokens itself).
- **`workstream_strategy: parallel`** only when workstreams are genuinely independent (e.g. ai-research *expand* ∥ *polish*); else `sequential`.
- Every `producers[].skills` entry must be a skill that actually exists — verify with `find-skills`, don't assume.
- **model** ∈ {cheap, standard, capable, max} (abstract; the concrete model is resolved per `references/platforms.md`). Back-compat: F1's Claude names (haiku/sonnet/opus/fable) are accepted and mapped. A missing model is filled by the rubric/profile default at dispatch.

## Capacity profile (plan parity)
Dreamteam runs on plans from a $20 tier (a below-capable conducting model, tight session limits, scarce capable-tier dispatch) to a $200 tier (headroom everywhere), and `run_policy`'s per-field defaults plus the reviewer tier floor are sized for the top plan — on a small plan the capable floor is unsatisfiable and the per-dispatch tier-unavailable fallback (`loop.md` §"Model tier + escalation") would flag every gate. The **capacity profile** — `low | standard | high` — sizes the run to the plan powering it. **Parity means the SAME verdict semantics and honesty bar at every plan: capacity changes SCHEDULING and TIER RESOLUTION, never the gate.**

- **Resolution.** Explicit `--capacity low|standard|high` → use it. Default **`auto`**: the CONDUCTING model's own tier is the best observable proxy for the plan — **below capable → `low` · capable → `standard` · max-tier → `high`** (what "resolves to a dispatchable model" means is per-host, `references/platforms.md`). This is a **heuristic — a proxy, overridable, never an oracle**: when it misreads the plan, the explicit flag corrects it. A **mid-run tier-unavailable event MAY downgrade the resolved profile ONCE** for the rest of the run, **flagged in the report** — never silently, never per-dispatch flip-flopping (and never an upgrade); a downgrade that lands `max_total_dispatches` at or below the dispatches already consumed trips the cap immediately — **STOP + escalate, per the cap's existing semantics, never a silent continue**.
- **Per-run defaults** (the hard ceilings above are UNCHANGED — still nothing overrides them):
  - **`low`** → `max_concurrent_agents` **3** · `max_total_dispatches` **30** · `budget_confirm_at` **15** · `workstream_strategy` biased **`sequential`** (genuine independence may still parallelize under the 3-cap) · the reviewer floor **resolves to the BEST AVAILABLE tier**, the substitution **declared ONCE at cast time as a "capacity note" line in the manifest `rationale`** — the note declares **ALL the run's expected capable→best-available resolutions (any capable-assigned role: the reviewer floor, a dispatched planner, an architecture/design lead), not the reviewer floor alone** (naming the resolved profile + the resolutions, e.g. "capacity low (auto: conductor below capable) — capable-assigned roles resolve capable→standard, the best this plan dispatches") — **replacing the per-dispatch tier-unavailable flagging for the whole run** (`loop.md`; the per-dispatch flag stays for surprises beyond the declared profile). Best-available means exactly that: **where capable IS dispatchable, best-available IS capable** — the substitution exists only where the plan lacks the tier.
  - **`standard`** → **5** · **45** · **20** · the rest as today.
  - **`high`** → today's defaults: **8** · **60** · **30**.
- **Guarantees.** **Capacity remaps SCHEDULING numbers and TIER RESOLUTION only. It NEVER weakens the gate: the Reality Checker is ALWAYS dispatched and its pass ALWAYS required; the risk-scaling guards and non-waivable force-includes hold (capacity never reclassifies a workstream's risk and never shrinks a panel below its class); `min_pass` semantics are unchanged; the evidence/honesty rules (`gate.md` §3) are unchanged.** The "Reviewers never drop below capable" floor is a statement about the tier ASSIGNED — capacity never assigns lower; on `low` the assigned floor RESOLVES to the best tier the plan actually has: **the BAR stays, the model behind it is the best available — declared, never hidden.**
- **Precedence + composition.** Capacity sets DEFAULTS; **explicit flags win** — `--models` / `--roster` (incl. `@tier`) / `--cost` override per §Precedence exactly as today. And capacity **composes with — never duplicates — the budget-aware scaling above**: capacity is the **defaults profile, resolved once at cast (moved after that only by the single flagged mid-run downgrade above)**; the live budget is a **run CEILING that only ever tightens further mid-run**. Both move scheduling only; neither touches the gate.

## Model tier (cheapest-that-fits)
Tiers are abstract — **cheap (haiku) → standard (sonnet) → capable (opus) → max (fable)** — and the concrete model is resolved per platform in `references/platforms.md`. Assign each dispatched role the cheapest tier that fits, per subagent-driven-development's signals:
- Mechanical producer (1–2 files, complete/clear spec, deterministic) → cheap.
- Integration / judgment producer (multi-file, pattern-matching, debugging) → standard.
- Planner (when a dispatched agent, not a skill), architecture/design lead, and ALL gate reviewers (Reality Checker especially) → capable (the default ceiling).
- Pre-fix **refuter** (`gate.md` §4) → standard — it executes one targeted evidence check, it does not judge the workstream; it is an evidence-producer, **not a reviewer**, so it sits below the reviewer floor without touching it ("Reviewers never drop below capable" stays literally true).
- max is NOT a default — top escalation rung / explicit override only.
- **Effort dial (platform-dependent, orthogonal to tier).** On platforms exposing a reasoning-effort lever (Codex `model_reasoning_effort`; Claude Code **Workflow-mode `agent()`**'s `effort` option — the Agent tool has none; `references/platforms.md`), a **cheap-tier mechanical producer MAY run at low effort**; **reviewers — Reality Checker especially — never below the platform default**. The dial **complements this rubric, never replaces it** — never a substitute for the right tier — **and it adjusts DOWNWARD only at dispatch**: on platforms where the `max` rung *is* capable + high effort (`references/platforms.md`), raising effort on a `capable` role **resolves to `max`** — the escalation ladder's top, taken only per `loop.md` §"Model tier + escalation" or an explicit override, never a casual dispatch-time raise.
Reviewers default capable regardless of producer tiers (the honesty backbone stays strong). Under a `low` **capacity profile** (§"Capacity profile") the assigned floor is unchanged — only its *resolution* lands on the best tier the plan actually dispatches, declared once at cast.

## The Caster agent prompt (hand this to a `general-purpose` subagent)
> You are the **Caster** for a `/dreamteam` run. Task: «TASK». Repo/domain context: «CONTEXT».
> Select a dreamteam to do this task — do **NOT** execute any of the work yourself.
> 1. Read the **available agents** from the Agent tool's agent-type list — use the live list, not memory.
> 2. Discover relevant **skills** with `find-skills` — don't assume a skill exists; check. Also consult `references/learnings.md` for prior learnings on this profile/task_kind — **the current project's entries (matching `project_key`) plus global ones** — and fold them in as overridable defaults weighted by their numeric `confidence` (cite the entry), not hard rules. Consult its **agent scouting ledger** under the same filters when choosing **between fitting candidates**: prefer the better first-try / lower-drop record, ignore a row with fewer than 3 dispatches as noise, and never touch the always-on Reality Checker. A row **never bans** an agent, and a **MISSING row is never held against one** — a never-dispatched agent competes on fit exactly as before. When the ledger sways a pick, cite the numbers in that rationale line.
> 3. Choose: a `planner` (agent or skill, or `null` if a plan is already supplied); optional `designers` as `{agent, model}`; `producers` as `{role, agent, model, skills[]}`; a `reviewers` panel (each as `{agent, model}`) that **ALWAYS includes Reality Checker** plus any domain reviewers the task demands (e.g. Security Engineer for auth/payments/infra; a methodology reviewer for research; Performance Benchmarker for load/latency work); `gate_policy {min_pass, max_fix_iterations}`; and `workstream_strategy` (`parallel` only if the workstreams are truly independent). Then assign each role a model tier per the Model-tier rubric (reviewers default capable); name the tier in each rationale line (e.g. "build→Mobile App Builder@standard: multi-file integration"). **Producers follow dreamteam's native minimal-code principle** — the simplest solution that fully works (YAGNI ladder), complete and correct, never cutting validation/security/accessibility — regardless of which skills are attached. For **code-producing** producers, also attach the composed principle-enforcers to their `skills[]` when installed (`find-skills`, each checked independently — proceed without any that's absent): `ponytail` (the minimal-code *enforcer*) and `karpathy-guidelines` (surface assumptions, surgical/minimal-diff changes, verifiable success criteria). They amplify dreamteam's native principles; the principles hold without them. For **navigating an existing codebase**, also attach the optional `graphify` AST code-graph as a **navigation aid** (`--graph on`/`auto`, `find-skills`/tool-gated, checked independently): add it to **dev/code-producing producers'** `skills[]` when installed, and surface the shared graph to the **Code Reviewer** as infra at dispatch (the manifest's `reviewers[]` carry no `skills[]`). Absent / `--graph off` → producer and reviewer read the tree directly and you record a `recommendations[]` entry — never a block. The graph is **infra, never a producer and never a gate**: any graph-derived claim that would gate a workstream is **verified against live code/tests** (`loop.md` §Graph, `gate.md` §2), never trusted from the graph.
> 4. Return **only** the dreamteam-manifest JSON above, with a one-line `rationale` per pick. No prose outside the manifest, no execution.

If a chosen agent/skill turns out to be unavailable, pick the closest alternate and say so in its rationale line (don't silently drop a role).

**Cast-time MCP inventory (availability only — rides the working set, never the preamble).** Dispatched units can already use connected MCP tools natively — what they lack is knowing what is AVAILABLE (a verifying reviewer that could drive a real browser never learns a Playwright MCP server is connected). So whichever path cast the manifest, at cast time the conductor **inventories the RELEVANT connected MCP capability** — **capability classes** (browser automation, docs lookup, database access, …), never an exhaustive tool dump — resolved per the **host's own MCP config** (`references/platforms.md`), and **hands it down in the WORKING SET** (`loop.md`) of the units it would help (a verifying reviewer gets *"browser automation available via the connected Playwright MCP server"*). **AVAILABILITY ONLY — never a dependency:** nothing is installed, no gate requires an MCP tool, and **absent MCP = silent skip — the unit works natively** (the graphify skip-if-absent posture, `loop.md` §Graph). Per-unit working-set content rides the brief's **TAIL**, never a new standing preamble brief (the assembly order below is untouched).

**Dispatch brief — the no-glazing standard (rides on every reviewer dispatch).** Whatever reviewer the manifest casts is **dispatched with dreamteam's objective / no-glazing standard in its brief**: state the unvarnished assessment, call severities by impact, and never praise-pad or soften a finding — a bare "looks good" with no specific finding is not a review (`gate.md` §3). **Vendored reviewer agent bodies are kept pristine** (`THIRD_PARTY_NOTICES.md`), so this standard rides on the **dispatch brief**, never the agent file — which is what lets it bind whatever agent is cast (vendored or host).

**Dispatch brief — evidence at emit time (rides on every reviewer dispatch).** Every reviewer is dispatched with the rule that **each finding it emits must carry its motivating evidence at emit time** — the verbatim `file:line`, the failing command + its output, or the reproducing case. A finding without that evidence is **unverified**, **does not count**, and is never a must-fix until re-emitted with evidence (`gate.md` §3, the emit-time admissibility rule — distinct from and complementary to §2's reconciliation-time adjudication). Like the no-glazing standard, this **rides on the dispatch brief, never the pristine agent body**, so it binds whatever reviewer is cast (vendored or host).

**Dispatch brief — confidence axis (rides on every reviewer dispatch).** Each reviewer also tags **every finding with a confidence** — how certain the claim is real, a *certainty* axis orthogonal to *severity*'s impact — so the gate surfaces the highest-signal findings first (`gate.md` §1–2). **This never weakens the honesty backbone:** confidence sets display order/emphasis only — a high-severity finding always surfaces regardless of confidence, the always-on Reality Checker always reports, and a genuine issue is never buried (`gate.md` §2 guard). This too rides on the dispatch brief, never the agent body.

**Dispatch brief — security method (rides on a security reviewer's dispatch).** When the manifest casts a reviewer to cover the **security dimension** (the `Security Engineer`, or any reviewer charged with security on auth/payments/infra/data-sensitive/AI surface), it is **dispatched with `references/security.md` as its method** — the stack-neutral OWASP/STRIDE checklist (secrets · supply-chain · CI/CD · OWASP Top-10 · STRIDE · LLM/AI · skill-supply-chain). Like the no-glazing and evidence briefs, this **rides on the dispatch brief, never the pristine agent body** (`THIRD_PARTY_NOTICES.md`), so it binds whatever reviewer is cast (vendored or host). The method changes nothing about how findings enter the gate — they still carry emit-time evidence + severity + confidence (`gate.md` §1/§3) and the verdict stays `gate.md` §2; it is a method the reviewer *may* follow, never a hard dependency.

**Dispatch brief — leaf firewall (rides on EVERY dispatch).** Every dispatched unit — producer, reviewer, refuter, planner, designer, Caster, or setup agent — is briefed: *"You are a **LEAF** for this dreamteam run. Do only your one assigned task and return your result. Do **NOT** invoke `/dreamteam`, do **NOT** orchestrate or act as a conductor, and do **NOT** dispatch or spawn any further subagents or Workflow scripts. dreamteam's session-stickiness and conductor rules do not apply to you."* Like the no-glazing / evidence / confidence / security briefs, this **rides on the dispatch brief, never the pristine agent body** (`THIRD_PARTY_NOTICES.md`), so it binds whatever unit is cast — vendored or host — and on every CLI, where no Workflow-harness nesting limit exists (`references/platforms.md`). It mirrors the `SKILL.md` `<SUBAGENT-STOP>` block, and the two are deliberately belt-and-suspenders over independent failure paths: (a) the block catches a unit that **auto-loads** the skill by description-match; (b) this brief catches a unit that **never loads** the skill but carries the brief.

**Dispatch brief — execution discipline (rides on every producer/reviewer dispatch).** Every producer/reviewer is dispatched with the rule that a shell command it runs MUST be **watched to completion** and its **output + exit status READ before proceeding** — never proceed on an unobserved or unfinished command. For a **long-running / backgrounded** command, use a **bounded** wait/poll (a timeout), then check the result; **never** sit in an open-ended wait — if it hangs past the bound, kill + retry ONCE or report the hang and continue with the evidence in hand, but **never** abandon the task sitting in a wait state (`loop.md` §"Execution discipline"; the operational half of the emit-time evidence rule, `gate.md` §3 — no observed output = no evidence = not done). This is a **discipline** rule — the harness owns the actual shell/poll, so this binds the model's obligation to observe it. Like the no-glazing / evidence / confidence / security / leaf-firewall briefs, it **rides on the dispatch brief, never the pristine agent body** (`THIRD_PARTY_NOTICES.md`), so it binds whatever unit is cast — vendored or host — on all 5 CLIs (`references/platforms.md`).

**Dispatch brief — report what you checked (rides on every reviewer dispatch).** Every reviewer's verdict must **name what it actually inspected** — the files read, the commands run, the `gate.md` §5 phases exercised — because an assessment that doesn't show its work is **inadmissible** (`gate.md` §"Verdict admissibility"). Like the no-glazing / evidence / confidence briefs, this **rides on the dispatch brief, never the pristine agent body** (`THIRD_PARTY_NOTICES.md`), so it binds whatever reviewer is cast (vendored or host).

**Dispatch-brief assembly — cache-stable preamble (ordering/stability only).** The standing briefs above are assembled as one **byte-identical PREAMBLE in one fixed order, FIREWALL-FIRST** — leaf firewall → no-glazing → evidence → confidence → security (conditional: when the manifest casts a security-covering reviewer it occupies this fixed slot; else the slot is simply absent) → execution discipline → report-what-you-checked — with **all per-leaf content** (role, task, success criterion, working set, context) in a **TAIL strictly after the preamble**. The order **deliberately leads with the identity firewall** — cache stability needs only *a* fixed order, and "placed first" is part of the firewall's compliance robustness (`SKILL.md` §Resilience) — so it **differs from the blocks' document order in this file**. **No brief's content changes** — each block keeps its exact text and its own stated scope (a reviewer-scoped rule binds only reviewers wherever it rides); this is an ordering/stability rule only. **The byte-identical claim is scoped (cost-only, never correctness):** the preamble is byte-identical across **all post-cast dispatches while the manifest's reviewer set is unchanged** — the Caster's own pre-manifest dispatch sits outside it (no manifest yet, so its security slot is unevaluable), and a setup-triggered **re-cast simply starts a new stable prefix** (it may add or remove the security-covering reviewer). Where the platform supports prefix caching, **provider-side caching hits that shared prefix across those dispatches**; where it lacks it, the ordering rule costs nothing.

## Autonomy (the manifest is always printed first)
- **`auto`** (default): print the dreamteam + rationale → **proceed** to the loop.
- **`confirm`**: print the dreamteam + rationale → **wait for the user's OK** before producing.
- **Budget gate (auto-confirm on large fan-out).** Before producing, estimate the run's projected total dispatches (`run_policy` — `projected ≈ Σ workstreams × (producers + panel) × (1 + max_fix_iterations)`); if it crosses `run_policy.budget_confirm_at` (default 30 — the `high` capacity row; a resolved capacity remaps it, §"Capacity profile"), or `--depth exhaustive` / `--evolve` is set, **PRINT the projected fan-out + cost and WAIT for the user's OK — even under `--autonomy auto`.** Generalizes the existing exhaustive/evolve budget-print to any large run; auto-enters the `confirm` step above — no new machinery. In Workflow mode with a **user token target**, the projection also incorporates the harness's **live remaining-token budget** — per `run_policy`'s budget-aware scaling above.
