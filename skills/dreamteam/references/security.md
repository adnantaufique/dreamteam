# Dreamteam — Security Review Method (the security dimension's checklist)

A concise, **stack-neutral** method the **Security Engineer** — or whatever reviewer covers the security dimension — MAY follow when **security is in scope**. It turns the named role into a *method*, not just a name. It is a **reference/checklist the reviewer reads, NOT a tool, daemon, scanner, or vendored skill, and NOT a hard dependency**: absent, the reviewer still reviews; present, it gives phased coverage instead of leaving depth to whatever the cast agent happens to know.

**It feeds the normal gate — it adds no new machinery.** Every finding this method produces enters `gate.md` like any other finding: carrying its **motivating evidence at emit time** (the verbatim `file:line`, the failing command + its output, or the reproducing case), a **severity** (impact) and a **confidence** (certainty), per `gate.md` §1/§3. There is **no new gate step and no new verdict** — the verdict still comes from `gate.md` §2. Security findings are subject to the same honesty backbone: an unevidenced security claim is `unverified` and is never a must-fix until re-emitted with evidence.

**Reason always, verify when cheap** (the gate's evidence posture, `gate.md` §3). For each phase: **reason** about the risk from the diff/code/config every time; **verify** with a concrete check when that check is cheap and available (run the scanner, grep the history, pin-check the action, trace the input to its sink) — and **label the proof honestly** (`unit` / `integration` / `manual` / `unverified`), never faking a green. A phase that genuinely doesn't apply to the surface under review is **labelled N/A with a reason — never silently skipped** (mirrors `gate.md` §5).

**Scope-aware + infra-first.** Apply only the phases that fit the change — a diff touching UI copy doesn't need a full STRIDE pass. Work outermost-in (secrets → dependencies → CI/CD → application logic → AI surface) so an exposed secret or a poisoned dependency is caught before time goes into application-layer nuance.

## The phases

1. **Secrets archaeology.** Credentials, API keys, tokens, private keys, and connection strings in the **diff, the config/env files, and the git history** — plus weak/default secrets, secrets committed-then-rotated (still in history), and secrets logged or echoed. *Verify (cheap):* run a secret-scan over the diff — this is the same scan as `gate.md` §5 step 5, so **share that result, don't double-run**. A live secret is high-severity by default.

2. **Dependency / supply chain.** New or bumped dependencies: known CVEs, abandoned/unmaintained packages, **typosquats** and confusable names, lockfile integrity (no unpinned floating ranges where pinning is expected), risky transitive deps, and **install-time scripts** that execute on install. *Verify (cheap):* run the ecosystem audit (`npm`/`pnpm audit`, `pip-audit`, `osv-scanner`, `cargo audit`, …), diff the lockfile, and confirm each added package's identity/source.

3. **CI/CD configuration.** The pipeline as an attack surface: `pull_request_target` (or equivalent) that checks out and runs **untrusted fork code with secrets in scope**; **unpinned actions/steps** (pin to a commit SHA, not a moving tag); over-broad token/permission scopes; secrets exposed to forks or leaked into logs; unsafe self-hosted-runner exposure. *Verify (cheap):* read the workflow files; check action pins and the `permissions:`/token scope.

4. **OWASP Top-10 (application layer).** Walk the changed surface for: injection (SQL / command / template / NoSQL); **broken access control & authorization** (missing or weak authz on a new endpoint, route, or resource); broken authentication / session handling; SSRF; insecure deserialization; security misconfiguration; sensitive-data exposure (crypto at-rest/in-transit, PII handling); XSS / output-encoding; and unvalidated input crossing a trust boundary. *Verify (cheap):* trace one untrusted input to its sink, reason about an exploit, and write/point to a test where the check is cheap.

5. **STRIDE per component / trust boundary.** For each component or boundary the change touches, reason through **S**poofing (identity), **T**ampering (integrity), **R**epudiation (auditability / logging), **I**nformation disclosure (confidentiality), **D**enial of service (availability), and **E**levation of privilege (authorization). *Verify (cheap):* name the boundary explicitly and the control sitting on it — a boundary with no control is a finding.

6. **LLM / AI security** (when the surface includes models, prompts, agents, tools, or RAG). **Prompt injection** from untrusted or ingested content; **treat model output as untrusted** (it can carry XSS / command / SQL into a downstream sink); **tool/function-call authorization** (an agent's tools are an exec surface — least privilege, no unbounded auto-confirm); data exfiltration via prompts; secret / training-data leakage; and **excessive agency** (the model doing more than the task needs). *Verify (cheap):* confirm resurfaced or ingested text is neutralized as **data, not instructions** before it reaches the model, and that tool calls are scoped and human-gated where they should be (consistent with `SKILL.md` §Autonomy's User-Challenge gate).

7. **Skill / extension supply chain** (when the change installs or trusts a skill, plugin, agent, or MCP server). Installed extensions are an attack surface — Snyk's *ToxicSkills* audit found a substantial share of skills carry malicious or risky behavior. Verify **identity & source** (owner/repo@skill matches the approved candidate — **no substitution or typosquat**), a **pinned** install command carrying **no `-y`/auto-confirm**, least privilege, and that the skill body was reviewed before it is trusted. *Verify (cheap):* this **reuses dreamteam's existing `setup` gate** — the same `(a) present · (b) identity == approved · (c) pinned command, no -y` checks in `recommend.md` and `gate.md` §3; this phase simply generalizes that paranoia to any trusted extension.
